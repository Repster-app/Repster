# Work Packages: Set + Stats Services

**Inputs**: Design documents from `kitty-specs/004-set-and-stats-services/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, contracts/SetServiceProtocol.swift, contracts/StatsServiceProtocol.swift
**Tests**: No automated tests (manual testing for v1 per constitution).

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package must be independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `kitty-specs/004-set-and-stats-services/tasks/` generated by `/spec-kitty.tasks`.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- All source file paths are relative to `Reppo/` (the Xcode project root).

## Path Conventions
- **Service protocols**: `Reppo/Core/Services/Protocols/`
- **Service implementations**: `Reppo/Core/Services/`
- **Repository protocols (existing)**: `Reppo/Core/Repositories/Protocols/`
- **Repository implementations (existing)**: `Reppo/Core/Repositories/`
- **Models (existing, read-only)**: `Reppo/Data/Models/`
- **Enums (existing, read-only)**: `Reppo/Data/Enums/`
- **Extensions (existing)**: `Reppo/Core/Extensions/UnitConversion.swift`
- **App entry point**: `Reppo/App/ReppoApp.swift`

## Work Package WP02: Repository Aggregation Methods (Priority: P0)

**Goal**: Add Core Data `NSFetchRequest` + `NSExpression` aggregation methods to `SetRepository` for database-level SUM/MAX/COUNT. These are the cold-path methods used by `StatsService.rebuildAll()` to recompute stats without loading sets into memory — per specdoc S8.6 "let the database do aggregation work."
**Independent Test**: `fetchAggregateStats()` returns correct totalSets/totalReps/totalVolume/maxWeight for a known dataset. `fetchWorkoutCount()` returns distinct workout count. `fetchBestE1RM()` returns the highest e1RM.
**Prompt**: `kitty-specs/004-set-and-stats-services/tasks/WP02-repository-aggregation.md`
**Estimated prompt size**: ~450 lines

### Included Subtasks
- [x] T006 [P] Add `fetchAggregateStats()`, `fetchWorkoutCount()`, `fetchBestE1RM()` method signatures to `SetRepositoryProtocol`
- [x] T007 [P] Create `SetAggregateResult` value type in protocol file if not already present (may be in WP01's StatsServiceProtocol file — if so, just reference it)
- [x] T008 Implement `fetchAggregateStats()` in `SetRepository` using Core Data `NSFetchRequest` + `NSExpression`
- [x] T009 Implement `fetchWorkoutCount()` in `SetRepository` using Core Data count distinct
- [x] T010 Implement `fetchBestE1RM()` in `SetRepository` using SwiftData sort+fetchLimit(1)

### Implementation Notes
- **Critical**: `fetchAggregateStats()` uses Core Data's `NSFetchRequest<NSDictionary>` with `NSExpressionDescription` for `sum:`, `max:`, `count:` — producing real SQL aggregation at the SQLite level
- Bridge from SwiftData to Core Data: Access the underlying `NSManagedObjectContext` from `modelContext` — requires `import CoreData` in the repository
- `fetchWorkoutCount()` counts distinct `workoutId` values for sets of a given exercise — use `NSExpression(forFunction: "count:", arguments: [NSExpression(forKeyPath: "workoutId")])` with `returnsDistinctResults = true`
- `fetchBestE1RM()` uses standard SwiftData `FetchDescriptor` with sort by e1RM DESC + fetchLimit(1) — no Core Data needed
- All aggregation methods accept `excludeWarmups: Bool` and always exclude partial sets per spec
- The `fetchTotalVolume()` method already exists in SetRepository but uses Swift iteration — the new `fetchAggregateStats()` supersedes it for rebuild scenarios

### Parallel Opportunities
- T006 (protocol additions) and T007 (value type) are independent
- T008/T009/T010 (implementations) are independent methods but all depend on T006

### Dependencies
- Depends on WP01 (SetAggregateResult value type must exist).

### Risks & Mitigations
- **Core Data bridge**: The `modelContext` to `NSManagedObjectContext` bridge may differ between SwiftData versions. In `@ModelActor` actors, `modelContext` is the standard access. The underlying Core Data context can be accessed via the container. Test thoroughly.
- **NSExpression for volume**: `SUM(effectiveWeight * reps)` as a single expression may require `NSExpression(forFunction: "multiply:by:", ...)` composition. Alternative: compute volume from separate SUM(effectiveWeight) and SUM(reps) if multiplication is problematic.
- **Entity name in NSFetchRequest**: The Core Data entity name for SwiftData models may differ from the class name. Use the `NSManagedObjectModel` to verify.

## Work Package WP03: StatsService — Incremental Updates (Priority: P1) 🎯 MVP

**Goal**: Implement StatsService with the incremental `updateStats(for:event:)` method that runs on the hot path after every set save/edit/delete. This is the write-time stats update that keeps ExerciseStats current without scanning history — per specdoc S8.4.
**Independent Test**: After saving a set via StatsService.updateStats(.save), ExerciseStats shows correct totalSets, totalReps, totalVolume. After editing, stats reflect the delta. After deleting, stats are decremented. Partial sets are excluded from volume. Warmup sets excluded when setting is off.
**Prompt**: `kitty-specs/004-set-and-stats-services/tasks/WP03-stats-incremental.md`
**Estimated prompt size**: ~500 lines

### Included Subtasks
- [x] T011 Create `StatsService` actor skeleton with repository dependencies via initializer injection
- [x] T012 Implement private eligibility helper: `shouldCountForStats(setType:, hasData:, excludeWarmups:) -> Bool`
- [x] T013 Implement `updateStats(for:event:)` — `.save` case: create or increment ExerciseStats
- [x] T014 Implement `updateStats(for:event:)` — `.edit` case: compute delta, adjust stats
- [x] T015 Implement `updateStats(for:event:)` — `.delete` case: decrement stats, re-query maxWeight if needed

### Implementation Notes
- StatsService is a plain Swift `actor` (NOT `@ModelActor`) — same pattern as PRService
- Dependencies injected via init: `ExerciseStatsRepositoryProtocol`, `SetRepositoryProtocol`, `ExerciseRepositoryProtocol`, `HealthProfileRepositoryProtocol`
- **Hot path**: All operations use pure arithmetic on the cached ExerciseStats object. No database aggregation queries. O(1).
- `.save` case: fetch existing ExerciseStats (or create new), increment totalSets/totalReps/totalVolume/maxWeight/lastPerformedDate
- `.edit` case: compute delta between old and new values, adjust totals. If maxWeight might have changed, re-query via `setRepo.fetchMaxEffectiveWeight()`
- `.delete` case: decrement totals. If deleted set had maxWeight, re-query for new max. If no sets remain, consider deleting ExerciseStats.
- Volume = effectiveWeight × reps. Partial sets always excluded. Warmups excluded when `HealthProfile.includeWarmupsInVolume == false`.
- `totalWorkouts`: On save, check if this is first set for exercise in this workout. On delete, check if any sets remain for exercise in this workout.

### Parallel Opportunities
- T011 (skeleton) must come first
- T012 (eligibility helper) is independent
- T013/T014/T015 are independent cases of the same method

### Dependencies
- Depends on WP01 (StatsServiceProtocol and value types must exist).

### Risks & Mitigations
- **maxWeight re-query on delete**: Only needed when deleted set's effectiveWeight equals current stats.maxWeight. Use existing `setRepo.fetchMaxEffectiveWeight(for:reps:)` — but note this filters by reps. For overall maxWeight across all reps, may need a new overload or fetch all reps and take max.
- **totalWorkouts accuracy**: Incrementing on save and decrementing on delete requires knowing if other sets for this exercise exist in the same workout. Use `setRepo.fetchSets(for workoutId:)` filtered by exerciseId — small query.
- **ExerciseStats save semantics**: When updating an existing stats object, we mutate its properties and call `exerciseStatsRepo.save()`. SwiftData may auto-save on property mutation within the same ModelContext — verify that `save()` doesn't create duplicates.

## Work Package WP04: StatsService — Rebuild + SetService (Priority: P1)

**Goal**: Implement StatsService rebuild methods (`rebuild(for:)`, `rebuildAll()`) and the complete SetService actor with save/edit/delete orchestration including effectiveWeight computation and the PR + stats pipeline.
**Independent Test**: `rebuildAll()` produces identical ExerciseStats to incremental updates for the same dataset (SC-004). SetService.save() computes effectiveWeight correctly for bodyweightFactor=0.65 and stores it. The full pipeline (persist → effectiveWeight → PR eval → stats) completes within 100ms.
**Prompt**: `kitty-specs/004-set-and-stats-services/tasks/WP04-rebuild-and-set-service.md`
**Estimated prompt size**: ~600 lines

### Included Subtasks
- [x] T016 Implement `rebuild(for:)` in StatsService — single exercise stats recompute using `fetchAggregateStats()`
- [x] T017 Implement `rebuildAll()` in StatsService — iterate all exercises, call `rebuild(for:)` each
- [x] T018 Create `SetService` actor skeleton with 6 dependencies via initializer injection
- [x] T019 Implement `computeEffectiveWeight()` private helper — specdoc S5.4 formula
- [x] T020 Implement `save()` — full sequential pipeline: effectiveWeight → persist → PR → stats
- [x] T021 Implement `edit()` — capture old values, recompute, re-trigger pipeline
- [x] T022 Implement `delete()` — capture values, delete, PR recompute, stats decrement

### Implementation Notes
- **StatsService.rebuild(for:)**:
  1. Delete existing ExerciseStats for exercise (if exists)
  2. Call `setRepo.fetchAggregateStats(exerciseId, excludeWarmups, excludePartial: true)` — Core Data NSExpression
  3. Call `setRepo.fetchWorkoutCount(for: exerciseId)` and `setRepo.fetchBestE1RM(for: exerciseId)`
  4. Create new ExerciseStats with aggregated values, save
- **StatsService.rebuildAll()**: Read HealthProfile once, fetchAll exercises, call rebuild(for:) per exercise
- **SetService** is a plain Swift `actor` with 6 injected dependencies: `SetRepositoryProtocol`, `ExerciseRepositoryProtocol`, `BodyweightEntryRepositoryProtocol`, `HealthProfileRepositoryProtocol`, `PRServiceProtocol`, `StatsServiceProtocol`
- **computeEffectiveWeight()**: `effectiveWeight = weight + (closestBodyweight × exercise.bodyweightFactor)`. If bodyweightFactor == 0 → effectiveWeight = weight. If no bodyweight entry → effectiveWeight = weight (warn user).
- **save() pipeline** (sequential, specdoc S4.5):
  1. Compute effectiveWeight, store on set
  2. `setRepo.save(set)` — immediate persistence (FR-012)
  3. `prService.evaluate(...)` — using set's primitives (UUID, Double, Int, etc.)
  4. `statsService.updateStats(for: exerciseId, event: .save(...))` (FR-003)
  5. Return `SetSaveResult`
- **edit()**: Fetch old values first, recompute effectiveWeight, save, call `prService.evaluateAfterEdit()`, call `statsService.updateStats(.edit(old, new))`
- **delete()**: Capture all values before deletion, delete via repo, call `prService.handleDeletion()`, call `statsService.updateStats(.delete(...))`
- **FR-010**: effectiveWeight is never recalculated retroactively. Historical sets keep their value. `edit()` recalculates because the set itself is being edited.

### Parallel Opportunities
- T016/T017 (rebuild) and T018-T022 (SetService) are in different files, can proceed in parallel
- Within SetService: T018 (skeleton) must come first, T019 (effectiveWeight) must come before T020-T022

### Dependencies
- Depends on WP02 (repository aggregation methods for rebuild) and WP03 (StatsService skeleton + updateStats for SetService to call).

### Risks & Mitigations
- **Core Data aggregation results**: NSExpression may return NSDecimalNumber or NSNumber — handle type conversion carefully in `rebuild(for:)`.
- **effectiveWeight with nil weight**: If `set.weight` is nil, effectiveWeight should be nil. The set won't have `hasData=true` unless it has actual values. Handle nil safely.
- **SetService ↔ PRServiceProtocol**: SetService calls PRServiceProtocol methods. Since we code against the protocol only (not the real PRService from 003), this compiles even without 003 merged. At runtime, the real PRService is injected via ServiceContainer.
- **100ms budget**: All operations local SwiftData. Estimated: ~5ms persist + ~5ms effectiveWeight + ~10ms PR eval + ~5ms stats = ~25ms. Well within budget.

**Goal**: Create or update `ServiceContainer` to hold SetService and StatsService alongside PRService. Wire into `ReppoApp.swift` via SwiftUI environment. Verify the full app builds with zero errors.
**Independent Test**: App launches. ServiceContainer holds all three services. SetService is accessible from the SwiftUI environment. Build succeeds with zero errors.
**Prompt**: `kitty-specs/004-set-and-stats-services/tasks/WP05-service-container-wiring.md`
**Estimated prompt size**: ~300 lines

### Included Subtasks
- [x] T023 Create `ServiceContainer` class (or update existing from 003) with PRService + StatsService + SetService
- [x] T024 Wire `ServiceContainer` into `ReppoApp.swift` alongside `RepositoryContainer`
- [x] T025 Verify full project build — zero errors, all protocols satisfied

### Implementation Notes
- **ServiceContainer**: `@Observable final class` that takes `RepositoryContainer` in init. Creates StatsService first (no service deps), then PRService (no service deps), then SetService (depends on both PRService and StatsService).
- Initialization order matters: StatsService → PRService → SetService (SetService depends on both)
- If 003 has already been merged and `ServiceContainer` exists, extend it. If not, create it fresh.
- Wire into SwiftUI via `.environment(serviceContainer)` in ReppoApp
- **Build verification**: Every protocol conformance must be satisfied. All `async throws` methods implemented. All value types Sendable.

### Parallel Opportunities
- T023 and T024 are sequential (container must exist before wiring)
- T025 is a verification step after both

### Dependencies
- Depends on WP04 (all service implementations must exist for wiring).

### Risks & Mitigations
- **Feature 003 merge state**: ServiceContainer may or may not exist from 003. The prompt must handle both cases (create new or extend existing).
- **PRService concrete type**: If 003 hasn't merged, we need a placeholder/stub PRService to satisfy the ServiceContainer. Alternative: make ServiceContainer accept protocols, not concrete types. This is better for testability anyway.
- **SwiftUI environment**: Both RepositoryContainer and ServiceContainer should be injected. Pattern: `.environment(repositoryContainer).environment(serviceContainer)` in the app body.
