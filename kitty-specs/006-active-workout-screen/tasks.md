# Work Packages: Active Workout Screen

**Inputs**: Design documents from `kitty-specs/006-active-workout-screen/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, contracts/ActiveWorkoutViewModelContract.swift
**Tests**: No automated tests (manual testing for v1 per constitution).

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package must be independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `kitty-specs/006-active-workout-screen/tasks/` generated by `/spec-kitty.tasks`.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- All source file paths are relative to `Reppo/` (the Xcode project root).

## Path Conventions
- **Views**: `Reppo/Features/Workout/Views/`
- **View components**: `Reppo/Features/Workout/Views/Components/`
- **ViewModels**: `Reppo/Features/Workout/ViewModels/`
- **Extensions**: `Reppo/Core/Extensions/`
- **App root**: `Reppo/App/`
- **Services (existing, read-only)**: `Reppo/Core/Services/`
- **Models (existing, read-only)**: `Reppo/Data/Models/`
- **Enums (existing, read-only)**: `Reppo/Data/Enums/`

## Work Package WP01: Design Tokens + Atomic Components (Priority: P0)

**Goal**: Create the design token Color extensions and the four atomic UI components that all subsequent views compose. These are the building blocks — every other WP depends on them.
**Independent Test**: Each component renders correctly in Xcode preview with mock data. Color tokens match design-system.md hex values.
**Prompt**: `kitty-specs/006-active-workout-screen/tasks/WP01-design-tokens-and-atomic-components.md`
**Estimated prompt size**: ~400 lines

### Included Subtasks
- [x] T001 [P] Create Color extension with all design tokens per design-system.md (`bg`, `bgCard`, `bgInput`, `bgSubtle`, `bgHover`, `border`, `text`/`textPrimary`, `textMid`/`textSecondary`, `textDim`/`textTertiary`, `blue`, `blueSoft`, `green`, `greenSoft`/`successSoft`, `gold`, `goldSoft`, `red`)
- [x] T002 [P] Create `SetInputField` reusable numeric input component in `Reppo/Features/Workout/Views/Components/SetInputField.swift`
- [x] T003 [P] Create `PRBadgeView` in `Reppo/Features/Workout/Views/Components/PRBadgeView.swift`
- [x] T004 [P] Create `SetNumberBadge` in `Reppo/Features/Workout/Views/Components/SetNumberBadge.swift`
- [x] T005 [P] Create `CompletionCheckbox` in `Reppo/Features/Workout/Views/Components/CompletionCheckbox.swift`

### Implementation Notes
- Color extension goes in `Reppo/Core/Extensions/` (new file `DesignTokens.swift` or extend existing)
- All hex values from design-system.md Section 2 & 4. Dark mode only — no light mode variants.
- SetInputField: `bgInput` (#1F1F25) bg, 1px `border` (white 6%), focused state = `blue` border + blue tint bg, completed state = `green` at 6% bg + 15% border. Text: 16pt semibold, centered. Keyboard: `.decimalPad` for weight, `.numberPad` for reps/duration.
- PRBadgeView: Renders based on `CachedPRStatus?` — `.current` → gold badge (star + "PR"), `.matched` → blue badge ("="), nil/`.previous` → EmptyView.
- SetNumberBadge: 28×28pt, `bgSubtle`, 8pt radius, 13pt semibold. Warmup: italic "W", no bg. Completed: `green` fill, white checkmark SF Symbol.
- CompletionCheckbox: 26×26pt frame, 6pt radius. Unchecked: 2pt `textDim` border. Checked: `blue` fill, white checkmark. Tap area expanded to 44×44pt via `.contentShape`.

### Parallel Opportunities
- All 5 subtasks touch different files and are fully parallel.

### Dependencies
- None (starting package). Requires features 001-005 to exist on main.

### Risks & Mitigations
- Color naming: ensure the extension names don't collide with SwiftUI built-in colors (e.g., use `Color.appBlue` or `Color.dsBlue` prefix if needed).

## Work Package WP02: ActiveWorkoutViewModel — Core + Set Operations (Priority: P0)

**Goal**: Build the complete ViewModel with all state management, service integration, set operations, and exercise operations. After this WP, all service-calling logic exists and views can bind to it.
**Independent Test**: Create VM with mock services, call `loadActiveWorkout()`, verify workout/exercises/sets populated. Call `completeSet()`, verify SetService.save() invoked and local state updated with PR result.
**Prompt**: `kitty-specs/006-active-workout-screen/tasks/WP02-active-workout-viewmodel.md`
**Estimated prompt size**: ~500 lines

### Included Subtasks
- [x] T006 Create `ActiveWorkoutViewModel` skeleton — `@Observable @MainActor final class` with all properties, dependencies, initializer
- [x] T007 Implement `loadActiveWorkout()` — check for active workout via WorkoutService, load exercises and sets, group sets by exerciseId
- [x] T008 Implement `completeSet()` — collect input values, call SetService.save(), update local state with SetSaveResult (cachedPRStatus, effectiveWeight), trigger rest timer start
- [x] T009 Implement `addSet()` and `addWarmupSet()` — create new WorkoutSet with correct orderInExercise/orderInWorkout, persist via SetService, append to local state
- [x] T010 Implement `deleteSet()` and `changeSetType()` — call SetService.delete()/edit(), update local state, handle PR badge side effects from `PREvaluationResult.affectedSetIds`
- [x] T011 Implement exercise operations — `addExercises()` (from picker), `removeExercise()` (with set cascade), `reorderExercises()` (update exercise order array)

### Implementation Notes
- ViewModel is `@Observable @MainActor final class` — not an actor. Services are actors, called with `await` from `Task {}` blocks.
- Dependencies: `WorkoutServiceProtocol`, `SetServiceProtocol`, `ExerciseServiceProtocol` — injected via initializer.
- `loadActiveWorkout()`: call `workoutService.getActiveWorkout()` → if nil, this means no active workout (the FAB/exercise picker creates one). If exists, fetch exercises by looking at distinct exerciseIds from the workout's sets.
- `completeSet()` is the critical hot path: build WorkoutSet → `setService.save()` → read `SetSaveResult.prResult.newStatus` → update set's `cachedPRStatus` in local dict → update any `affectedSetIds` (other sets whose badge changed) → start rest timer.
- `addExercises()`: called from picker sheet with selected exerciseIds. For each: fetch exercise, add to `exercises` array, create initial empty working set.
- `removeExercise()`: delete all sets for that exercise in this workout via service, remove from local arrays.
- `reorderExercises()`: update `exercises` array order. No immediate persistence needed — order is implicit until next save.

### Parallel Opportunities
- T006 must come first. T007-T011 are somewhat sequential (each builds on the VM state) but T011 (exercise ops) is independent of T008-T010 (set ops).

### Dependencies
- Depends on WP01 (needs RestTimerState type from WP01, though technically the enum could be defined here).

### Risks & Mitigations
- **SetService.save() contract**: Verify the exact method signature on SetServiceProtocol — the save method takes a WorkoutSet and returns SetSaveResult. The VM must construct the WorkoutSet correctly with all required fields.
- **Concurrency**: All service calls are `async` — wrap in `Task {}`. Don't block the main thread. Use `isLoading` flag during initial load.
- **Side effects from PR pipeline**: `PREvaluationResult.affectedSetIds` may contain other sets whose `cachedPRStatus` changed. The VM must update those in local state too.

**Goal**: Build the complete set table — the primary interaction surface. Renders an adaptive grid of set rows with input fields, badges, and checkboxes, plus Add Set/Warmup buttons and long-press context menus. Covers User Story 2.
**Independent Test**: Render SetTableView with mock data for a WEIGHT_REPS exercise, verify 5-column layout. Switch to DURATION exercise, verify columns adapt. Verify warmup row opacity, completed row green tint, PR badge rendering.
**Prompt**: `kitty-specs/006-active-workout-screen/tasks/WP03-set-table-ui.md`
**Estimated prompt size**: ~500 lines

### Included Subtasks
- [x] T012 Create `SetRowView` — single row composing SetNumberBadge + SetInputField(s) + PRBadgeView + CompletionCheckbox, with row height 52pt and 1px divider
- [x] T013 Create `SetTableView` — header row with column labels, ForEach currentSets → SetRowView, column widths per design-system (Set 42pt, Weight 1fr, Reps 1fr, PR 44pt, Check 40pt)
- [x] T014 Implement column adaptation — switch on `exercise.trackingType` to show correct input fields (WEIGHT_REPS → weight+reps, DURATION → duration, WEIGHT_DISTANCE → weight+distance, WEIGHT_REPS_DURATION → weight+reps+duration)
- [x] T015 Implement warmup and completed row styling — warmup rows at 0.5 opacity, completed rows with `successSoft` green tint background, completed set badge with green check
- [x] T016 [P] Add [+ Add Set] and [+ Add Warmup] buttons below the set table
- [x] T017 Add long-press context menu on set rows — "Edit Set Type" (submenu with SetType cases) and "Delete Set" (with destructive role)

### Implementation Notes
- SetRowView receives a `WorkoutSet` binding + the `Exercise` (for trackingType) + callbacks for completion and input changes.
- Column layout: Use `HStack` with `.frame(width:)` for fixed columns (Set 42pt, PR 44pt, Check 40pt) and flexible `.frame(maxWidth: .infinity)` for input columns.
- Row dividers: `.overlay(Divider().foregroundColor(.white.opacity(0.03)), alignment: .bottom)` or equivalent.
- Column adaptation (T014): This is critical — the input fields shown depend on `exercise.trackingType`. Create a helper function `columnsForTrackingType()` per plan.md.
- Warmup row (T015): Apply `.opacity(0.5)` on the entire row when `set.setType == .warmup`.
- Completed row (T015): Apply `Color.green.opacity(0.06)` background, change set badge to green check.
- Context menu (T017): Use `.contextMenu { }` modifier on SetRowView. "Edit Set Type" can be a nested menu or a sheet with SetType picker.

### Parallel Opportunities
- T016 (add buttons) is independent of the row/table views.
- T012-T015 are sequential (row → table → adaptation → styling).

### Dependencies
- Depends on WP01 (atomic components: SetInputField, PRBadgeView, SetNumberBadge, CompletionCheckbox).
- Depends on WP02 (ViewModel provides data bindings).

### Risks & Mitigations
- **Column layout flexibility**: Different trackingTypes show different numbers of input columns. Ensure the flexible column widths (`1fr`) adapt correctly when there are 1, 2, or 3 input fields.
- **Performance**: Use `LazyVStack` inside a `ScrollView` for the set list to handle exercises with many sets (though typically <20 per exercise).

## Work Package WP04: ExerciseTabStripView — Tab Navigation (Priority: P1) MVP

**Goal**: Build the horizontal scrollable tab strip for switching between exercises. Supports tap to select, long-press to delete, and drag to reorder. Covers User Story 3.
**Independent Test**: Render tab strip with 3 exercises, tap to switch active tab, verify styling changes. Long-press to see delete option. Drag to reorder.
**Prompt**: `kitty-specs/006-active-workout-screen/tasks/WP04-exercise-tab-strip.md`
**Estimated prompt size**: ~350 lines

### Included Subtasks
- [x] T018 Create `ExerciseTabStripView` — horizontal `ScrollView` with tab buttons for each exercise
- [x] T019 Implement tab styling — active tab: `blue` bg + white text + 8pt radius; inactive tab: `bgCard` bg + `textDim` text + 8pt radius
- [x] T020 Add long-press context menu on tabs — "Delete Exercise" with confirmation alert
- [x] T021 Implement drag-to-reorder gesture — custom `DragGesture` with position-based reorder and animation

### Implementation Notes
- Tab strip is a `ScrollView(.horizontal, showsIndicators: false)` containing an `HStack` of tab buttons.
- Each tab shows `exercise.name`, truncated if too long. Use `.lineLimit(1)` and padding.
- Active tab detection: bind `selectedExerciseIndex` to the ViewModel.
- Long-press delete (T020): Use `.contextMenu` modifier. When "Delete Exercise" tapped, show a confirmation `Alert` before calling `viewModel.removeExercise(at:)`.
- Drag reorder (T021): This is the most complex subtask. Use a custom `DragGesture` on each tab. Track the dragged tab's offset, calculate the new position based on x-coordinate, and call `viewModel.reorderExercises()` on drop. Apply `.animation(.default)` for smooth visual reorder.
- Alternative fallback for drag: If the custom drag gesture proves too complex, fall back to a long-press context menu with "Move Left" / "Move Right" options.

### Parallel Opportunities
- T018-T019 (basic strip + styling) are sequential. T020 (long-press) and T021 (drag) are independent of each other.

### Dependencies
- Depends on WP01 (design tokens for colors).
- Depends on WP02 (ViewModel for exercise list and selection state).

### Risks & Mitigations
- **Drag gesture conflicts**: Custom DragGesture on tab buttons may conflict with ScrollView's built-in scroll gesture. Mitigate with `.simultaneousGesture` or by requiring a long-press to activate drag (LongPressGesture → DragGesture sequence).
- **ScrollView auto-scroll**: When the active tab changes, the ScrollView should auto-scroll to show the active tab. Use `ScrollViewReader` + `.scrollTo()`.

## Work Package WP06: Rest Timer (Priority: P2)

**Goal**: Build the rest timer that auto-starts when a set is completed, counting down from `exercise.defaultRestTime`. Supports +30s extension and manual dismiss. Covers User Story 4.
**Independent Test**: Complete a set for an exercise with defaultRestTime=180, verify timer starts at 3:00 and counts down. Tap +30s, verify 30s added. Tap Dismiss, verify timer hidden. Complete another set, verify timer restarts.
**Prompt**: `kitty-specs/006-active-workout-screen/tasks/WP06-rest-timer.md`
**Estimated prompt size**: ~350 lines

### Included Subtasks
- [x] T027 Create `RestTimerState` enum (`idle`, `running(remaining:total:)`, `finished`) and timer management methods in ViewModel (`startRestTimer`, `addTime`, `dismissTimer`)
- [x] T028 Implement timer countdown logic — `Timer.publish(every: 1)` subscription, decrement remaining, handle reaching zero (transition to `.finished`)
- [x] T029 Create `RestTimerView` — countdown display (MM:SS format), circular or linear progress indicator, [+30s] button, [Dismiss] button
- [x] T030 Wire auto-start and edge cases — trigger rest timer from `completeSet()`, dismiss on new set completion start, recalculate remaining time when app returns from background

### Implementation Notes
- RestTimerState (T027): Define as a simple enum in the ViewModel file or a separate helper file. The VM owns all timer state.
- Timer countdown (T028): Use `Timer.publish(every: 1, on: .main, in: .common).autoconnect()`. Store the `AnyCancellable` subscription. On each tick, decrement remaining. When remaining <= 0, set state to `.finished`. Store the timer start `Date` so background recalculation works.
- RestTimerView (T029): Positioned below the set table or as a floating bar. Show MM:SS countdown, optional circular progress ring (`remaining/total` ratio). Two action buttons: "+30s" calls `viewModel.addTime(30)`, "Dismiss" calls `viewModel.dismissTimer()`. Use design tokens: `bgCard` background, `blue` for progress, `textPrimary` for time.
- Auto-start (T030): In `completeSet()`, after successful save, call `startRestTimer(duration: currentExercise.defaultRestTime ?? 90)`. If a timer is already running when a new set is completed, restart the timer. On `scenePhase` change from `.background` to `.active`, recalculate remaining from `Date()` vs stored start time.

### Parallel Opportunities
- T029 (RestTimerView UI) can be built independently from T028 (timer logic) if given a mock state.

### Dependencies
- Depends on WP02 (ViewModel where timer state and methods live).

### Risks & Mitigations
- **Timer accuracy after backgrounding**: iOS suspends timers when backgrounded. Mitigation: store the absolute `timerStartDate` and total duration. On foreground return, compute `remaining = total - Date().timeIntervalSince(timerStartDate)`. If negative, set to `.finished`.
- **Combine subscription leak**: Must cancel the timer subscription in `dismissTimer()` and when VM is deinitialized.

## Work Package WP07: Workout Summary Sheet — Finish Flow (Priority: P1) MVP

**Goal**: Build the finish workout flow — summary sheet showing workout stats, notes input, RPE selector, and "Save & Close" action. Covers User Story 5.
**Independent Test**: Tap "Finish Workout", verify summary sheet appears with correct totals (date, duration, sets, volume, PRs). Enter notes and select RPE, tap "Save & Close", verify workout status=completed and screen dismisses.
**Prompt**: `kitty-specs/006-active-workout-screen/tasks/WP07-workout-summary-sheet.md`
**Estimated prompt size**: ~400 lines

### Included Subtasks
- [x] T031 Create `WorkoutSummaryData` and `ExerciseSummary` types + `computeSummary()` method on ViewModel
- [x] T032 Create `WorkoutSummarySheet` view — date header, duration, total volume, total sets, exercise list with set counts and best lifts, PRs highlighted with gold badge
- [x] T033 Add notes `TextEditor` and RPE selector (1-10) to summary sheet
- [x] T034 Implement `finishWorkout(notes:perceivedEffort:)` in ViewModel — call WorkoutService.finishWorkout(), update workout with notes and RPE
- [x] T035 Handle post-finish navigation — dismiss summary sheet, dismiss active workout screen, navigate back (Calendar tab placeholder for now)

### Implementation Notes
- WorkoutSummaryData (T031): Computed from local ViewModel state — `setsByExercise` dict. Total sets = count of sets where `hasData == true`. Total volume = sum of `effectiveWeight * reps` for completed sets. Per-exercise: count sets, find best weight. PRs hit = count sets where `cachedPRStatus == .current`.
- Summary sheet (T032): Presented via `.sheet(isPresented: $showFinishSheet)`. Layout: `ScrollView` with `VStack` — date/duration header, stats row (total sets, total volume), exercise list (each with name, set count, best lift), PR highlight section. Use design tokens: `bgCard` for cards, `gold` for PR highlights.
- Notes + RPE (T033): `TextEditor` for free-form notes with `bgInput` styling. RPE selector: either a horizontal row of 1-10 buttons or a `Picker`. Each RPE value is tappable, selected one highlighted with `blue`.
- finishWorkout (T034): Call `workoutService.finishWorkout(workoutId)`. The WorkoutService handles setting status=completed, endTime, duration. The VM also needs to pass notes and perceivedEffort — check if WorkoutService.finishWorkout() accepts these or if they need to be set separately via workout update.
- Post-finish (T035): After `finishWorkout()` succeeds, dismiss the sheet, then dismiss the active workout view. Navigation target: Calendar tab. For now, just dismiss back to ContentView.

### Parallel Opportunities
- T031 (data types) and T032 (sheet UI) can be built in parallel if T032 uses mock data.
- T033 (inputs) is independent of T031-T032.

### Dependencies
- Depends on WP02 (ViewModel finishWorkout method and state access).
- Depends on WP05 (ActiveWorkoutView must exist to present the sheet from).

### Risks & Mitigations
- **WorkoutService.finishWorkout() API**: The existing method signature may not accept notes/perceivedEffort. Check the protocol — if it only sets status and endTime, the VM may need to update the workout object's notes/perceivedEffort fields separately before calling finish.
- **Navigation after finish**: Dismissing a `.fullScreenCover` AND navigating to Calendar requires coordination. May need a shared `NavigationState` or `@Environment(\.dismiss)` chain.

## Dependency & Execution Summary

```
WP01 (Design Tokens + Atomic Components)
  ↓
WP02 (ViewModel)
  ↓
  ├── WP03 (Set Table UI) ── ┐
  ├── WP04 (Tab Strip) ────── ├── WP05 (Screen Assembly + Navigation)
  ├── WP06 (Rest Timer)       │
  └── WP07 (Summary Sheet) ── ┘
```

- **Sequence**: WP01 → WP02 → {WP03, WP04, WP06, WP07 in parallel} → WP05
- **Parallelization**: After WP02, four work packages can run concurrently (WP03, WP04, WP06, WP07) since they touch different files.
- **MVP Scope**: WP01 + WP02 + WP03 + WP04 + WP05 + WP07 constitute the MVP. WP06 (rest timer) is P2 and can be deferred.
- **Critical path**: WP01 → WP02 → WP03 → WP05 (the set table is the core interaction).
