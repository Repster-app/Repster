# Work Packages: Workout + Exercise + Bodyweight Services

**Inputs**: Design documents from `kitty-specs/005-workout-exercise-bodyweight-services/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, contracts/WorkoutServiceProtocol.swift, contracts/ExerciseServiceProtocol.swift, contracts/BodyweightServiceProtocol.swift
**Tests**: No automated tests (manual testing for v1 per constitution).

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package must be independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `kitty-specs/005-workout-exercise-bodyweight-services/tasks/` generated by `/spec-kitty.tasks`.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- All source file paths are relative to `Reppo/` (the Xcode project root).

## Path Conventions
- **Service protocols**: `Reppo/Core/Services/Protocols/`
- **Service implementations**: `Reppo/Core/Services/`
- **Repository protocols**: `Reppo/Core/Repositories/Protocols/`
- **Repository implementations**: `Reppo/Core/Repositories/`
- **Models (existing, read-only)**: `Reppo/Data/Models/`
- **Enums (existing, read-only)**: `Reppo/Data/Enums/`
- **DI container**: `Reppo/Core/Services/ServiceContainer.swift`

## Work Package WP01: Protocols, Error Types + Repository Additions (Priority: P0)

**Goal**: Establish all contracts (service protocols, error types) and add the 5 new repository methods needed for cascade deletion. This is the foundation that enables all subsequent service implementation.
**Independent Test**: All protocols, error types, and repository methods compile. Repository methods can be called (even if not yet exercised by services).
**Prompt**: `kitty-specs/005-workout-exercise-bodyweight-services/tasks/WP01-protocols-and-repository-additions.md`
**Estimated prompt size**: ~400 lines

### Included Subtasks
- [x] T001 [P] Add 3 new methods to `SetRepositoryProtocol` and implement in `SetRepository` (`deleteSets(for workoutId:)`, `deleteSets(forExercise:)`, `fetchExerciseIds(for workoutId:)`)
- [x] T002 [P] Add `deleteAll(for exerciseId:)` to `PerformanceRecordRepositoryProtocol` and implement in `PerformanceRecordRepository`
- [x] T003 [P] Add `fetch(byId:)` to `BodyweightEntryRepositoryProtocol` and implement in `BodyweightEntryRepository`
- [x] T004 [P] Create `WorkoutServiceError`, `ExerciseServiceError`, `BodyweightServiceError` error enums
- [x] T005 [P] Create `WorkoutServiceProtocol` in `Reppo/Core/Services/Protocols/WorkoutServiceProtocol.swift`
- [x] T006 [P] Create `ExerciseServiceProtocol` in `Reppo/Core/Services/Protocols/ExerciseServiceProtocol.swift`
- [x] T007 [P] Create `BodyweightServiceProtocol` in `Reppo/Core/Services/Protocols/BodyweightServiceProtocol.swift`

### Implementation Notes
- Contract sources of truth: `kitty-specs/005-workout-exercise-bodyweight-services/contracts/` (3 protocol Swift files)
- Repository additions are simple SwiftData operations: predicate-based batch delete, predicate-based fetch + map to UUID set
- Error enums are small (2-3 cases each), defined in each service's protocol file
- All protocols conform to `Sendable` and declare all methods as `async throws`
- `Reppo/Core/Services/Protocols/` directory already exists from feature 003

### Parallel Opportunities
- All 7 subtasks touch different files and are fully parallel
- T001/T002/T003 are repository work; T004-T007 are service protocol work — zero overlap

### Dependencies
- None (starting package). Requires features 001-004 to exist on main.

### Risks & Mitigations
- `SetRepository.deleteSets()` must use SwiftData batch delete with predicate, not fetch-then-delete-each. If SwiftData doesn't support batch delete directly, fetch all matching then delete in a loop within the same `modelContext.save()` call.
- `fetchExerciseIds()` needs to return `Swift.Set<UUID>` (not `Set` — the `WorkoutSet` naming rule).

## Work Package WP02: BodyweightService + WorkoutService (Priority: P1) MVP

**Goal**: Implement both BodyweightService (thin CRUD wrapper) and WorkoutService (workout lifecycle + cascade deletion). After this WP, users can start/finish workouts, log bodyweight, and delete workouts with proper cascade.
**Independent Test**: Start a workout, verify status=inProgress. Start again, verify same workout returned. Finish workout, verify status=completed and duration set. Delete workout, verify sets removed and PRs/stats rebuilt for affected exercises.
**Prompt**: `kitty-specs/005-workout-exercise-bodyweight-services/tasks/WP02-bodyweight-and-workout-services.md`
**Estimated prompt size**: ~500 lines

### Included Subtasks
- [x] T008 Create `BodyweightService` actor with initializer (2 repository deps)
- [x] T009 Implement BodyweightService CRUD: `saveEntry()`, `updateEntry()`, `deleteEntry()`, `fetchAllEntries()`
- [x] T010 Implement `BodyweightService.closestBodyweight(to:)` delegating to repository
- [x] T011 Create `WorkoutService` actor with initializer (4 deps: workoutRepo, setRepo, prService, statsService)
- [x] T012 Implement `WorkoutService.startWorkout()` with active workout check + `getActiveWorkout()`
- [x] T013 Implement `WorkoutService.finishWorkout()` with status/endTime/duration update
- [x] T014 Implement WorkoutService read methods: `fetchWorkout()`, `fetchWorkouts(for:)`, `fetchAllWorkouts()`
- [x] T015 Implement `WorkoutService.deleteWorkout()` cascade: fetchExerciseIds → bulk delete sets → delete workout → rebuild per exercise

### Implementation Notes
- BodyweightService (T008-T010) is a thin wrapper — each method is 3-6 lines plus healthProfile association
- WorkoutService.startWorkout() must check `workoutRepo.fetchInProgress()` first; if exists, return it (no error, no duplicate — user decision)
- WorkoutService.finishWorkout() calculates duration as `Int(endTime.timeIntervalSince(startTime))` seconds
- deleteWorkout() cascade is the most complex method: must get affected exerciseIds BEFORE deleting sets, then rebuild PRs/stats for each affected exercise
- Both services are plain `actor` types (not `@ModelActor`), matching the pattern from PRService/SetService/StatsService

### Parallel Opportunities
- T008-T010 (BodyweightService) and T011-T015 (WorkoutService) are independent services, fully parallel

### Dependencies
- Depends on WP01 (protocols and repository methods).

### Risks & Mitigations
- Cascade deletion must call `prService.rebuild(for:)` and `statsService.rebuild(for:)` per exercise — these are existing methods from features 003/004. Verify they exist before implementing.
- `startWorkout()` returning existing workout: ensure the returned workout object is usable by the caller (same ModelContext considerations as other repository fetches).

## Work Package WP03: ExerciseService + ServiceContainer Wiring (Priority: P1) MVP

**Goal**: Implement ExerciseService (the most complex service with trackingType immutability enforcement, metadata mutability rules, and cascade deletion) and wire all three new services into ServiceContainer. After this WP, feature 005 is complete.
**Independent Test**: Create exercise, verify persisted. Update trackingType with no sets, verify allowed. Add sets (via existing SetService), attempt trackingType change, verify blocked. Change bodyweightFactor, verify rebuild triggered. Delete exercise, verify all sets/stats/PRs cleaned up. Build compiles with zero errors.
**Prompt**: `kitty-specs/005-workout-exercise-bodyweight-services/tasks/WP03-exercise-service-and-wiring.md`
**Estimated prompt size**: ~500 lines

### Included Subtasks
- [x] T016 Create `ExerciseService` actor with initializer (6 deps: exerciseRepo, setRepo, exerciseStatsRepo, performanceRecordRepo, prService, statsService)
- [x] T017 Implement `ExerciseService.createExercise()` and basic read methods (`fetchExercise`, `fetchAllExercises`, `searchExercises`, `exerciseHasSets`)
- [x] T018 Implement `ExerciseService.updateExercise()` with trackingType immutability enforcement (specdoc S5.6)
- [x] T019 Implement rebuild-required field change detection in `updateExercise()` (bodyweightFactor, unilateral, bilateralLoadFactor, equipmentType)
- [x] T020 Implement `ExerciseService.deleteExercise()` cascade: delete sets → delete ExerciseStats → delete PerformanceRecords → delete exercise
- [x] T021 Update `ServiceContainer` to create and expose WorkoutService, ExerciseService, BodyweightService
- [x] T022 Verify build compiles with zero errors — all protocols satisfied, no missing dependencies

### Implementation Notes
- `updateExercise()` is the most nuanced method: must fetch current persisted exercise to compare field values, then decide if rebuild needed
- The `originalTrackingType` parameter is passed by the caller (ViewModel will snapshot it before editing). The service compares against it for the immutability check.
- For rebuild-required field detection: compare `exercise.bodyweightFactor != persisted.bodyweightFactor` etc. The service must fetch the persisted version to detect changes.
- `deleteExercise()` cascade does NOT need rebuild — everything for this exercise is removed (sets, stats, PRs, exercise itself). Different from workout deletion which affects shared exercises.
- ServiceContainer init order: StatsService → PRService → SetService → BodyweightService → WorkoutService → ExerciseService. BodyweightService has no service deps. WorkoutService and ExerciseService both depend on PRService + StatsService.

### Parallel Opportunities
- T016-T020 (ExerciseService implementation) must be sequential (each builds on the actor)
- T021 (ServiceContainer) depends on all services existing
- T022 (build verification) is final

### Dependencies
- Depends on WP01 (protocols and repository methods) and WP02 (BodyweightService and WorkoutService implementations needed for ServiceContainer wiring).

### Risks & Mitigations
- **Rebuild-required field detection**: Fetching the persisted exercise adds one extra repository call per update. This is acceptable — exercise updates are rare (not on the hot path).
- **ServiceContainer growing**: Now 6 services in one init. Still clean and manageable. If it grows beyond ~10, consider splitting into sub-containers (not needed for v1).
- **Exercise deletion ordering**: Must delete sets FIRST (they reference exerciseId), then stats/PRs, then exercise. If order is wrong, foreign key constraints could block deletion.

## Dependency & Execution Summary

- **Sequence**: WP01 → WP02 → WP03
- **Parallelization**: Within WP01, all 7 subtasks are parallel (different files). Within WP02, BodyweightService and WorkoutService are parallel.
- **MVP Scope**: All 3 WPs constitute the full feature. WP01+WP02 delivers workout lifecycle and bodyweight. WP03 completes exercise management.
- **Critical path**: WP01 (foundation) is the blocker. Once WP01 is done, WP02 and WP03 could theoretically run in parallel IF ServiceContainer wiring is deferred — but sequential is safer.
