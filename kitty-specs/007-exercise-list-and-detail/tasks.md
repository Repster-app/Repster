# Work Packages: Exercise List + Detail

**Inputs**: Design documents from `/kitty-specs/007-exercise-list-and-detail/`
**Prerequisites**: plan.md (required), spec.md (user stories), research.md, data-model.md, contracts/view-contracts.md, quickstart.md
**Tests**: Manual testing only (no automated tests per constitution).

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package is independently deliverable and testable.

**Prompt Files**: Each work package references a matching prompt file in `kitty-specs/007-exercise-list-and-detail/tasks/` generated by `/spec-kitty.tasks`.

## Subtask Format: `[Txxx] [P?] Description`
- **[P]** indicates the subtask can proceed in parallel (different files/components).
- Includes precise file paths relative to `Reppo/`.

## Work Package WP02: Exercise Card & List Components (Priority: P0)

**Goal**: Build the reusable `ExerciseCardView`, `MuscleFilterStrip`, `SortOptionMenu`, supporting types, and the `ExerciseListViewModel` that powers the list.
**Independent Test**: Components render correctly in isolation with mock data. ViewModel loads exercises, filters by muscle group, sorts, and manages search state.
**Prompt**: `kitty-specs/007-exercise-list-and-detail/tasks/WP02-exercise-card-and-list-components.md`

### Included Subtasks
- [x] T006 Create `ExerciseListViewModel` with search, filter, sort, and selection state in `Reppo/Features/Exercise/ViewModels/ExerciseListViewModel.swift`
- [x] T007 [P] Create `MuscleFilterStrip` component in `Reppo/Features/Exercise/Views/Components/MuscleFilterStrip.swift`
- [x] T008 [P] Create `SortOptionMenu` component in `Reppo/Features/Exercise/Views/Components/SortOptionMenu.swift`
- [x] T009 [P] Create `ExerciseCardView` in `Reppo/Features/Exercise/Views/ExerciseCardView.swift`
- [x] T010 Create supporting types (`WorkoutHistoryGroup`, `ExerciseChartData`) in `Reppo/Features/Exercise/Models/`

### Implementation Notes
- `ExerciseListViewModel` fetches all exercises + their `ExerciseStats` on load, then filters/sorts client-side.
- `MuscleFilterStrip`: horizontal `ScrollView` of pill toggles. Derive unique muscle groups from loaded exercises via `exercises.compactMap { $0.primaryMuscle }`. Multiple can be active.
- `SortOptionMenu`: SwiftUI `Menu` with 3 options (A-Z, Most Recent, Most Used). Sort by `exercise.name`, `stats.lastPerformedDate`, `stats.totalWorkouts`.
- `ExerciseCardView`: Uses `AppCard` pattern from design-system.md (bgCard, 14pt radius, 14pt padding). Shows name, primaryMuscle, equipmentType, trackingType, lastPerformedDate, maxWeight/bestE1RM.

### Parallel Opportunities
- T007, T008, T009 are independent component files and can proceed in parallel.

### Dependencies
- Depends on WP01 (needs enums and directory structure).

### Risks & Mitigations
- `ExerciseStats` may be nil for exercises with no workout history. Card must handle nil stats gracefully (show "--" or "Never" for last performed).

## Work Package WP03: Exercise List View & Dual Mode (Priority: P1) MVP

**Goal**: Assemble `ExerciseListView` with search bar, filter strip, sort options, and exercise cards. Implement browse mode (tap -> detail) and selection mode (tap -> toggle, "Start Workout (N)" button).
**Independent Test**: Open Exercise List, search by name, filter by muscle group, sort by A-Z/recent/used. In browse mode, tap card placeholder. In selection mode, select 3 exercises, verify count badge and "Start Workout (3)" button.
**Prompt**: `kitty-specs/007-exercise-list-and-detail/tasks/WP03-exercise-list-view-and-dual-mode.md`

### Included Subtasks
- [x] T011 Create `ExerciseListView` assembling search bar, filter strip, sort menu, and scrollable card list in `Reppo/Features/Exercise/Views/ExerciseListView.swift`
- [x] T012 Implement browse mode: tap card triggers NavigationLink to ExerciseDetailView (stub destination until WP04)
- [x] T013 Implement selection mode: tap toggles selection, "Start Workout (N)" / "Add (N)" button at bottom
- [x] T014 Implement empty state (zero exercises) with message and [+ New] prompt
- [x] T015 Implement no-results state for search/filter with "No exercises found" message

### Implementation Notes
- Use `.searchable(text: $viewModel.searchText, prompt: "Search exercises")` on the `NavigationStack` list.
- Browse mode vs selection mode determined by `ExerciseListMode` init parameter.
- In browse mode, "Start Workout (N)" appears when `viewModel.hasSelection` is true (user can also select exercises in browse mode to start a workout, per spec).
- In `.addToWorkout` mode, presented as sheet. Uses `onExercisesSelected` callback to return selected IDs.
- Bottom action button: fixed at bottom with `safeAreaInset(edge: .bottom)`.
- [+ New] button in toolbar (`.toolbar { ToolbarItem(placement: .topBarTrailing) }`) or as floating action.

### Parallel Opportunities
- T014 (empty state) and T015 (no-results) can proceed in parallel.

### Dependencies
- Depends on WP02 (needs ExerciseListViewModel, ExerciseCardView, MuscleFilterStrip, SortOptionMenu).

### Risks & Mitigations
- Two modes in one view may create conditional complexity. Keep mode-specific behavior isolated to small switch blocks, not spread across the view.

## Work Package WP04: Exercise Detail View (Priority: P1) MVP

**Goal**: Build the reusable `ExerciseDetailView` with History, PRs, and Charts tabs. This component is used in 3 contexts (Exercise List browse, Calendar, Active Workout).
**Independent Test**: Push Exercise Detail from a stub, verify History tab shows past sessions newest first, PRs tab shows suffix-max filtered table, Charts tab shows e1RM trend and volume charts.
**Prompt**: `kitty-specs/007-exercise-list-and-detail/tasks/WP04-exercise-detail-view.md`

### Included Subtasks
- [x] T016 Create `ExerciseDetailViewModel` with lazy-loaded history, PRs, and chart data in `Reppo/Features/Exercise/ViewModels/ExerciseDetailViewModel.swift`
- [x] T017 Create `ExerciseDetailView` shell with exercise header and History/PRs/Charts tab picker in `Reppo/Features/Exercise/Views/ExerciseDetailView.swift`
- [x] T018 [P] Create `ExerciseHistoryView` showing past sessions grouped by workout in `Reppo/Features/Exercise/Views/ExerciseHistoryView.swift`
- [x] T019 [P] Create `ExercisePRsView` rendering suffix-max PR table in `Reppo/Features/Exercise/Views/ExercisePRsView.swift`
- [x] T020 [P] Create `ExerciseChartsView` with e1RM LineMark and volume BarMark in `Reppo/Features/Exercise/Views/ExerciseChartsView.swift`

### Implementation Notes
- `ExerciseDetailViewModel` loads data per-tab on demand (not all at once). `loadHistory()`, `loadPRs()`, `loadCharts()` called when tabs become active.
- History: Fetch sets via `SetRepository.fetchSets(for exerciseId:)`, group by `workoutId` into `WorkoutHistoryGroup`, sort newest first. Each group shows date + set rows (reuse `PRBadgeView` for PR indicators).
- PRs: Call `PRService.fetchPRTable(for: exerciseId)` which returns suffix-max filtered `[PRTableEntry]`. Render as a table with Reps, Weight (with unit conversion), Date columns.
- Charts: e1RM trend = `LineMark(x: .value("Date", point.date), y: .value("e1RM", point.value))`. Volume per session = `BarMark(...)`. Use `Color.accent` for chart marks. Fetch last 20 sessions of data.
- Tab picker: Use a custom segmented picker or pill strip matching the `ExerciseTabStripView` visual pattern.
- The view takes `exerciseId: UUID` only. It creates its own ViewModel internally per research.md decision R3.

### Parallel Opportunities
- T018 (History), T019 (PRs), T020 (Charts) are independent view files and can proceed in parallel.

### Dependencies
- Depends on WP01 (needs enums and directory structure).

### Risks & Mitigations
- History tab may have many sessions. Use `LazyVStack` for efficient rendering and consider limiting initial load to last 20 sessions with "Load More" option.
- PR table could be empty for new exercises. Show an appropriate empty state.

## Work Package WP05: Create/Edit Exercise Sheet (Priority: P1)

**Goal**: Build the Create/Edit Exercise form sheet with all fields from specdoc Section 6.3, including trackingType immutability enforcement.
**Independent Test**: Tap [+ New], fill form, save, verify exercise created. Edit existing exercise, verify trackingType locked when sets exist.
**Prompt**: `kitty-specs/007-exercise-list-and-detail/tasks/WP05-create-edit-exercise-sheet.md`

### Included Subtasks
- [x] T021 Create `CreateEditExerciseViewModel` with form state, validation, and save logic in `Reppo/Features/Exercise/ViewModels/CreateEditExerciseViewModel.swift`
- [x] T022 Create `CreateEditExerciseSheet` UI with all exercise fields in `Reppo/Features/Exercise/Views/CreateEditExerciseSheet.swift`
- [x] T023 Implement trackingType lock: disable picker when `isTrackingTypeLocked` (exercise has sets)
- [x] T024 Wire save flow: create via `ExerciseService.createExercise()` or update via `ExerciseService.updateExercise()` with `onSave` refresh callback

### Implementation Notes
- Form fields (from specdoc Section 6.3): name (TextField, required), equipmentType (Picker), trackingType (Picker, lockable), primaryMuscle (TextField or Picker), secondaryMuscles (multi-select), movementPattern (Picker, optional), unilateral (Toggle), bilateralLoadFactor (TextField, optional), bodyweightFactor (TextField, default 0.0), weightIncrement (TextField, optional), defaultRestTime (TextField, optional, in seconds).
- Use SwiftUI `Form` with `Section` groupings for clean layout.
- Validation: name must not be empty. Show inline error indicators.
- `checkTrackingTypeLock()` calls `ExerciseService.exerciseHasSets(exerciseId)` on edit mode init.
- On save success: call `onSave?()` callback to refresh the exercise list, then dismiss the sheet.
- `updateExercise` requires `originalTrackingType` parameter for immutability enforcement in the service layer.

### Parallel Opportunities
- T021 (ViewModel) and T022 (View) can start in parallel, then integrate.

### Dependencies
- Depends on WP01 (needs directory structure and enums).

### Risks & Mitigations
- Many form fields could make the sheet feel cluttered. Group fields logically: Basic Info (name, equipment, tracking), Muscles (primary, secondary, pattern), Advanced (unilateral, bodyweight factor, increment, rest time).
- secondaryMuscles as a multi-select needs a custom picker or comma-separated input.

**Goal**: Retrofit `ActiveWorkoutView` with `[Sets] [History] [Charts]` sub-tabs per exercise, and replace the stub `ExercisePickerSheet` with `ExerciseListView(mode: .addToWorkout)`.

- Sub-tab picker: small segmented-style pill strip below the exercise tab strip. Use same visual tokens as `ExerciseTabStripView` (bgCard inactive, accent active).
- When `selectedSubTab == .sets`: show existing `SetTableView` (unchanged).
- When `selectedSubTab == .history`: show `ExerciseHistoryView(exerciseId: currentExercise.id, ...)`.

### Risks & Mitigations

- Deleting `ExercisePickerSheet.swift` removes the stub. Ensure the new `ExerciseListView(mode: .addToWorkout)` passes selected exercise IDs back to `ActiveWorkoutViewModel.addExercises()` correctly.

## Work Package WP07: Integration & Navigation Wiring (Priority: P2)

**Goal**: Connect all components with proper navigation: FAB -> Exercise List, browse tap -> Exercise Detail, "Start Workout" flow, [+ New] -> Create/Edit sheet, Edit/Delete from detail.
**Independent Test**: Full end-to-end flows: FAB -> browse -> detail -> back; FAB -> select 3 -> Start Workout -> Active Workout; Detail -> Edit -> save; Detail -> Delete -> confirm -> list refreshes.
**Prompt**: `kitty-specs/007-exercise-list-and-detail/tasks/WP07-integration-and-navigation.md`

### Included Subtasks
- [x] T030 Wire FAB button in ContentView to navigate to `ExerciseListView(mode: .browse)` via NavigationStack
- [x] T031 Wire browse-mode card tap in `ExerciseListView` to push `ExerciseDetailView(exerciseId:)`
- [x] T032 Wire "Start Workout" button: create workout via `WorkoutService`, navigate to `ActiveWorkoutView` with selected exercises
- [x] T033 Wire [+ New] button in `ExerciseListView` to present `CreateEditExerciseSheet(exercise: nil)`
- [x] T034 Wire Edit/Delete actions from `ExerciseDetailView` to `CreateEditExerciseSheet` and cascade delete confirmation

### Implementation Notes
- FAB navigation: The FAB overlay on ContentView's TabView should trigger a navigation path push on the active tab's NavigationStack. Alternatively, use a dedicated NavigationStack wrapping the tab content that the FAB pushes onto.
- "Start Workout" flow (T032): Call `WorkoutService.startWorkout()`, then for each selected exercise create initial empty sets via `SetService`, then present `ActiveWorkoutView` via fullScreenCover.
- Delete confirmation (T034): Use `.confirmationDialog` or `.alert` before calling `ExerciseService.deleteExercise()`. After delete, pop navigation back to list and refresh.
- Edit from detail: Present `CreateEditExerciseSheet(exercise: exercise)` via `.sheet`, with `onSave` callback that refreshes the detail view.
- Ensure navigation state is clean: popping from detail refreshes list, dismissing sheets refreshes relevant views.

### Parallel Opportunities
- T030 (FAB wiring) and T034 (edit/delete) target different files and can proceed in parallel.

### Dependencies
- Depends on WP03 (ExerciseListView), WP04 (ExerciseDetailView), WP05 (CreateEditExerciseSheet), WP06 (ActiveWorkoutView updates).

### Risks & Mitigations
- NavigationStack path management across tabs can be tricky. Consider using `@State private var navigationPath = NavigationPath()` per tab.
- "Start Workout" flow involves multiple service calls (create workout, add exercises, create sets). Handle errors gracefully -- if any step fails, clean up and show error.
- Active workout resume on launch must still work after ContentView rewrite. Test this path.
